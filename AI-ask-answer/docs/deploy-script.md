# Deploy скрипт (`test_deploy.sh`)

Описание как работи скриптът за деплой на сървъра: какво прави, режимите за тест и подобренията.

---

## Цел

Преди да качиш до прод на сървъра скриптът:

1. **Прави backup** на текущия работещ контейнер (като Docker образ).
2. **Спира** контейнерите.
3. **Изравнява** git repo с кода на сървъра (`git pull origin stage`).
4. **Стартира** нов билд с `docker-compose up -d --build`.
5. **Изпраща известие в Telegram** – успех или грешка + rollback.

Ако новият билд **не успее**, прод **не се чупи**: скриптът автоматично връща стария контейнер от backup (rollback) и отново изпраща известие в Telegram.

---

## Стъпки (какво прави скриптът)

| Стъпка | Действие |
|--------|----------|
| 1 | Проверява дали контейнерът `d-dimitrov` (или `d-dimitrov-local` при локален тест) работи. Ако не – стартира с `docker-compose up -d --build`. |
| 2 | Създава backup образ с `docker commit` и таг `d-dimitrov-backup:YYYY-MM-DD-HH-MM-SS`. |
| 3 | Спира контейнерите с `docker-compose down`. |
| 4 | Обновява кода с `git pull origin stage`. |
| 5 | Стартира нов билд с `docker-compose up -d --build`. |
| 6a | При **успех**: изпраща в Telegram „Деплой успешен“. |
| 6b | При **грешка**: прави rollback (виж по-долу), после изпраща „Деплой неуспешен. Възстановен от бекъп“. |

Всичко се записва и в лог файл в `deploy_logs/deploy_YYYY-MM-DD_HH-MM-SS.log`.

---

## Rollback (при грешка)

При неуспешен `docker-compose up -d --build` скриптът:

1. Спира контейнерите с `docker-compose down`.
2. Създава временен файл `docker-compose.rollback.yml`, в който сервизът `app` използва **backup образа** вместо да се билдва отново.
3. Стартира с **същия** compose конфиг (портове, volumes, env) – само образът е различен: `docker-compose -f docker-compose.yml -f docker-compose.rollback.yml up -d`.
4. Изтрива `docker-compose.rollback.yml`.

Така възстановеният контейнер е с **същите портове, volumes и env** като преди деплоя – прод не се чупи.

---

## Режими

### Нормален деплой (на сървъра)

```bash
./test_deploy.sh
```

- Контейнер: `d-dimitrov`
- Порт: 3400 (от `docker-compose.yml`)
- Използва се основният `docker-compose.yml`.

### Локален тест (да не пипаш прод)

```bash
LOCAL_TEST=1 ./test_deploy.sh
```

- Контейнер: **`d-dimitrov-local`** (отделен от прод).
- Порт: **3401** (не конфликтира с 3400).
- Използва се `docker-compose.yml` + `docker-compose.local.yml` (друга мрежа, `.env`, `./upload_local`).

Подходящ е за тест на скрипта на твоя машина преди да го пуснеш на сървъра. За локален тест трябва да имаш `.env` (напр. копиран от `.env.example`).

### Dry run (само показва командите)

```bash
DRY_RUN=1 ./test_deploy.sh
```

- **Нищо не се изпълнява** – не се прави `down`, `pull`, `up`.
- Печатат се само командите, които биха се изпълнили.

Удобно е за проверка какво ще направи скриптът преди реален деплой.

## Telegram

Скриптът изпраща известие в края:

- При **успех**: „Деплой успешен: &lt;име на контейнера&gt;“.
- При **грешка + rollback**: „Деплой неуспешен. Възстановен от бекъп: &lt;име на контейнера&gt;“.

За да работи, преди пускане трябва да са зададени:

- `TELEGRAM_BOT_TOKEN` – токенът на бота.
- `TELEGRAM_CHAT_ID` – chat ID (напр. от @userinfobot).

Пример:

```bash
export TELEGRAM_BOT_TOKEN="твоят_токен"
export TELEGRAM_CHAT_ID="твоят_chat_id"
./test_deploy.sh
```

Или на един ред:

```bash
TELEGRAM_BOT_TOKEN="..." TELEGRAM_CHAT_ID="..." ./test_deploy.sh
```

Ако не са зададени, скриптът работи нормално, само без известия в Telegram.

## Свързани файлове

| Файл | Описание |
|------|----------|
| `test_deploy.sh` | Основният deploy скрипт (в корена на проекта). |
| `docker-compose.yml` | Конфигурация за прод (контейнер `d-dimitrov`, порт 3400). |
| `docker-compose.local.yml` | Override за локален тест (контейнер `d-dimitrov-local`, порт 3401). |
| `deploy_logs/` | Директория с логове от всеки пуск (в `.gitignore`). |
| `docker-compose.rollback.yml` | Временно създаван при rollback, после изтриван (в `.gitignore`). |

---

## Промени и подобрения (история)

- **Backup**: Backup чрез `docker commit` с таг с дата/час; преди деплой винаги се прави backup на текущия контейнер.
- **Rollback чрез compose**: При грешка се възстановява с **същия** compose конфиг (портове, volumes, env), като се използва само backup образът – не обикновен `docker run` без конфигурация.
- **Telegram**: Опционални известия при успех и при грешка + rollback.
- **Локален тест**: `LOCAL_TEST=1` използва отделен контейнер и порт, за да не засяга прод.
- **Dry run**: `DRY_RUN=1` показва само командите без изпълнение.
- **Логове**: Целият изход отива в лог файл в `deploy_logs/`.
- **Локален compose**: `docker-compose.local.yml` използва локална мрежа (`d-dimitrov-local-net`), за да не зависи от външната `docker_network` на сървъра.

---

## Препоръчан ред за тест

1. Локално: `DRY_RUN=1 ./test_deploy.sh` – да видиш командите.
2. Локално: `LOCAL_TEST=1 ./test_deploy.sh` – пълен тест върху `d-dimitrov-local` и порт 3401.
3. На сървъра: `./test_deploy.sh` – реален деплой (с или без Telegram env променливи).
